## 2. "처음 코딩했을 때 스티키가 잘 됐다" — 그때 구조

- **원래** RichTextEditor는 **한 가지 형태만** 있었음:  
  루트 = `flex flex-col overflow-auto max-h-[60vh]`, 툴바 = `sticky top-0`.
- 스크롤은 **에디터 루트(60vh 박스) 안**에서만 일어남.
- `position: sticky`의 **nearest scroll ancestor** = 에디터 루트 → 툴바는 **60vh 박스 상단**에 붙음.
- 그때는 **article이 overflow-hidden이어도** 스티키가 잘 동작함.  
  (스크롤하는 쪽이 에디터 루트이지, article이 아니기 때문.)

## 3. 지금 구조에서 스티키가 어떻게 동작해야 하는지

### 내레이션 (noMaxHeight = true)

- 에디터 루트: `flex flex-col min-w-0 overflow-x-hidden` 만 있음 (overflow-y, max-h 없음).
- 스크롤 = **AdminPage 메인 콘텐츠 div**(overflow-y-auto).
- 스티키가 메인에 붙으려면, 툴바와 메인 사이에 **overflow: hidden인 조상이 없어야** 함.
- 그래서 article을 **overflow-visible**로 바꿔 둔 상태.

### 원문 AI 요약 (noMaxHeight = false)

- 에디터 루트: `overflow-x-hidden` + `overflow-y-auto max-h-[60vh]` → **원래와 동일**.
- 스크롤 = **에디터 루트 안** → 스티키의 scroll ancestor = 에디터 루트.
- 따라서 **원문 AI는 예전이랑 같은 방식**으로 툴바가 60vh 박스 상단에 붙어야 함.

**정리**:  
- "처음에 잘 됐다" = 에디터만 있을 때(60vh + overflow-auto) 스크롤·스티키가 **에디터 안**에서만 동작했기 때문.  
- 지금 코드도 **원문 AI**는 그 구조를 유지하고 있음. **내레이션**만 스크롤 주체가 메인으로 바뀌었고, 그걸 위해 article을 overflow-visible로 둔 상태.

## 4. "전혀 적용이 안 된다"일 때 가능한 원인 (진단만)

### 4-1. 스티키가 안 붙는 경우

- **가능성 1 (가장 큼)**: **배포/캐시**  
  - 실제 서비스에 올라간 JS가 **예전 빌드**임.  
  - 예전 코드에는 article이 `overflow-hidden`이고, noMaxHeight 분기·루트 overflow 조합이 지금과 다름.  
  - 그러면 스티키가 깨진 상태로 보일 수 있음.
- **가능성 2**: **article이 여전히 overflow-hidden**  
  - 리포에는 `overflow-visible`인데,  
    - 빌드 결과에 반영이 안 됐거나  
    - 다른 CSS(전역/레이아웃)가 article에 overflow를 덮어쓰고 있을 수 있음.
- **가능성 3**: **루트에 준 overflow-x-hidden**  
  - 우리가 루트에 **항상** `overflow-x-hidden`을 넣음.  
  - 스크롤 컨테이너는 overflow-y이므로, 이론상 스티키에는 영향 없어야 하나,  
    특정 브라우저/버전에서 sticky와 overflow 조합이 예상과 다르게 동작할 여지는 있음 (가능성은 낮음).

**확인 방법 (수정 없이)**  
- 배포된 페이지에서 개발자 도구 → **article** 선택 → computed **overflow** / **overflow-x** / **overflow-y** 확인.  
- **RichTextEditor 루트** div도 같은 방식으로 확인.  
  - 원문 AI: `overflow-y: auto`, `max-height: 60vh` 인지.  
  - 내레이션: overflow-y가 없고, 메인 콘텐츠 div가 overflow-y: auto 인지.

### 4-2. AI 분석에서 글이 회색 밖으로 나가는 경우

- **가능성 1**: **전역 word-break: keep-all 상속**
  - `src/frontend/src/index.css`에서 **body**(77–78줄)에 `word-break: keep-all`, `overflow-wrap: normal` 적용됨 (한글 잘림 금지).
  - 같은 파일에서 **.prose, .prose-lg**(223–227줄)에도 동일 적용. 에디터가 이 클래스 안에 있으면 여기서도 상속됨.
  - 이게 **상속**되어 contenteditable에도 적용됨.  
  - 에디터에만 `break-words` + `[word-break:break-word]`를 넣어서 **덮어쓰는** 구조가 맞는데, 적용이 안 되면 글은 밖으로 나감.
- **가능성 2**: **Tailwind arbitrary 클래스가 빌드에 없음**  
  - `[word-break:break-word]`는 Tailwind arbitrary property.  
  - content에 `./src/**/*.{js,ts,jsx,tsx}`가 있으므로 보통 포함되나,  
    빌드된 CSS에 `.\[word-break\:break-word\]` (또는 해당 유틸)가 실제로 있는지 확인 필요.
- **가능성 3**: **배포/캐시**  
  - 옛 JS/CSS가 로드되면 word-break 덮어쓰기가 없는 예전 코드가 실행됨.

**확인 방법 (수정 없이)**  
- 배포된 사이트에서 에디터 contenteditable div에 computed **word-break**, **overflow-wrap** 확인.  
  - 우리가 원하는 것: word-break: break-word (또는 유사), overflow-wrap: break-word.  
  - body 상속이면 word-break: keep-all일 수 있음.  
- 빌드 결과물 `public/assets/*.css`에서 `word-break` 검색해 arbitrary 클래스가 포함돼 있는지 확인.

## 5. 서버/배포 쪽 점검 (적용 여부와의 관계)

- 배포는 **push → GitHub Actions build-frontend → deploy job이 artifact 받아 FTP로 /html/**.
- 즉, **실제 서비스에 나가는 건 빌드 결과물**이고, 리포의 tsx만 바꿨다고 해서 바로 반영되는 게 아님.
- 다음이 한 번이라도 어긋나면 "우리만 수정했는데 적용 안 됨"이 됨:
  - push가 main에 안 올라감
  - build-frontend 실패
  - deploy job 실패 또는 FTP 미반영
  - 브라우저/CDN이 예전 index.html·JS/CSS 캐시 사용

**확인 순서 (수정 없이)**  
1. GitHub 저장소 **main** 최신 커밋이 우리가 수정한 커밋인지.  
2. **Actions** 탭에서 해당 push 기준 **Build Frontend** / **Deploy** 가 성공(녹색)인지.  
3. 배포된 서버 **/html/** 에서 index.html, assets/*.js·*.css **수정 시각**이 최근인지.  
4. 브라우저 **강력 새로고침(Ctrl+Shift+R)** 또는 시크릿 창으로 다시 테스트.  
5. Network 탭에서 로드되는 **JS 파일명(해시)** 이 최신 빌드의 해시와 같은지.
