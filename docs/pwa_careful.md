# PWA 완성 작업별 유저 영향 전수 검토

> 작성일: 2026-04-04
> 목적: PWA를 완성하여 Lighthouse/심사 통과 수준으로 올릴 때, 현재 40명의 유저 사용에 영향을 주는 요소를 사전 파악

## 현재 상태 요약

현재 PWA는 **의도적으로 3중 비활성화**되어 있습니다:
1. `sw.js` — 자기 해제 전용 (캐시 삭제 + unregister)
2. `index.html` 인라인 스크립트 — 기존 SW 전부 unregister (신규 등록 안 함)
3. `Layout.tsx` — `InstallPrompt` 컴포넌트 주석 처리

---

## 작업 1: `manifest.webmanifest`의 `display`를 `"browser"` → `"standalone"` 변경

| 항목 | 평가 |
|------|------|
| **영향도** | **없음** |
| **이유** | manifest는 브라우저가 "설치" 시에만 참조. 일반 웹 접속에서는 `display` 값이 UX에 아무 영향을 주지 않음 |
| **주의점** | 변경 후 크롬이 "설치 가능" 조건을 인식하면 주소창에 작은 설치 아이콘이 나타날 수 있음 (유저가 직접 누르지 않는 한 아무 일도 안 함) |

---

## 작업 2: 아이콘 PNG 파일 추가 (`icon-192.png`, `icon-512.png`, `apple-icon.png`)

| 항목 | 평가 |
|------|------|
| **영향도** | **없음** |
| **이유** | 정적 파일 추가일 뿐. 기존 HTML에서 이미 `<link>`로 참조하고 있지만 파일이 없어서 404임. 파일을 넣으면 404가 해소될 뿐 |
| **주의점** | `favicon-32.png`, `favicon-16.png`도 현재 파일 없음 → 같이 추가하면 좋음 |

---

## 작업 3: `sw.js`를 실제 캐싱 서비스 워커로 교체

| 항목 | 평가 |
|------|------|
| **영향도** | **중간 ~ 높음 (가장 주의 필요)** |

### 위험 요소 상세:

**A. SSE(Server-Sent Events) 스트리밍 간섭 — 위험도 높음**

현재 다음 기능들이 `fetch` + `ReadableStream`으로 SSE 스트리밍을 사용합니다:
- `article-chat.php` — 기사 챗봇 (ArticleChatPanel, AIWorkspace)
- `search-analysis.php` — 지능형 검색 종합 분석 (SearchPage)
- `admin/chat-stream.php` — 관리자 채팅

SW의 `fetch` 핸들러가 이 스트리밍 요청을 가로채면 **응답이 완료될 때까지 버퍼링**되거나, 캐시된 응답을 반환해 **실시간 스트리밍이 깨질 수 있습니다.**

**대응:** SW `fetch` 핸들러에서 `/api/` 경로 전체를 **network-only**로 설정하거나, `Accept: text/event-stream` 헤더가 있는 요청을 무조건 패스스루해야 합니다.

**B. API 응답 캐싱 — 위험도 높음**

뉴스 목록, 기사 상세, 인증(`/api/auth/me`) 등이 모두 `/api/` 경로입니다. SW가 이를 캐시하면:
- 로그인/로그아웃 상태가 꼬임
- 새 기사가 안 보임
- 구독 상태가 갱신 안 됨

**대응:** `/api/**` 경로는 반드시 **network-only** (캐시 절대 금지).

**C. `useVersionCheck` 훅 충돌 — 위험도 중간**

현재 5분마다 `/version.json?t=timestamp`를 `cache: 'no-store'`로 fetch합니다. SW가 이를 가로채서 캐시된 버전을 반환하면 **배포 후 자동 새로고침이 작동 안 합니다.**

**대응:** `version.json` 요청도 network-only.

**D. 오프라인 폴백 페이지 — 위험도 낮음**

오프라인 시 빈 화면 대신 "인터넷 연결을 확인하세요" 같은 폴백을 보여주는 것은 **유저 경험 개선**이므로 부정적 영향 없음.

**E. 앱 셸(index.html) 캐싱 — 위험도 중간**

앱 셸을 precache하면 재방문 속도가 빨라지지만, 배포 후 구버전 셸이 남아 있을 수 있습니다. `useVersionCheck`가 이미 5분 간격으로 버전을 확인하지만, SW가 index.html을 캐시하면 이 체크 자체가 구버전 JS로 돌아갈 수 있습니다.

**대응:** Workbox의 `stale-while-revalidate` 전략 + `skipWaiting` + `clientsClaim`을 써서 새 SW 등록 즉시 활성화.

## 작업 4: `index.html`에서 SW 등록 코드 복원

| 항목 | 평가 |
|------|------|
| **영향도** | **낮음** |
| **이유** | SW 등록 자체는 백그라운드에서 일어남. 유저가 느끼는 것은 없음 |
| **주의점** | 기존 "해제 전용 sw.js"가 이미 등록된 유저 40명의 브라우저에 캐시되어 있을 수 있음. 새 sw.js를 배포하면 브라우저가 바이트 비교 후 자동 업데이트하므로 문제없지만, **등록 코드가 먼저 배포되고 sw.js는 아직 구버전**인 타이밍 문제가 생길 수 있음 |
| **대응** | sw.js 파일을 먼저 교체한 뒤, 다음 배포에서 등록 코드를 복원 (또는 같은 커밋에서 함께) |

## 작업 5: `InstallPrompt` 컴포넌트 주석 해제

| 항목 | 평가 |
|------|------|
| **영향도** | **낮음 (UX 변화만)** |
| **이유** | 이미 완성된 컴포넌트. 활성화하면 유저에게 하단 설치 배너가 나타남 |
| **주의점** | `shouldShowInstallUI()`가 이미 standalone 모드 체크, 하루 스누즈, "다시 보지 않기" 등을 처리하므로 과도한 노출은 없음 |
| **이전에 "다시 보지 않기"를 누른 유저는?** | localStorage에 `pwa_install_never_prompt=1`이 있으면 안 나옴. 하지만 이 컴포넌트가 지금까지 비활성이었으므로 이 값이 세팅된 유저는 없을 것 → **모든 유저에게 1회 나타남** |

---

## 작업 6: `.htaccess` 관련

| 항목 | 평가 |
|------|------|
| **영향도** | **없음** |
| **이유** | 현재 `.htaccess`에서 `manifest.webmanifest`는 JSON 확장자로 접근 차단 규칙(`FilesMatch "\.(json|...)$"` → `Require all denied`)에 걸리지 않음 (`.webmanifest` 확장자). sw.js도 `.js` 파일이라 정적 파일로 정상 서빙. 수정 불필요 |

---

## 종합 위험 매트릭스

| 작업 | 위험도 | 유저 체감 | 비고 |
|------|--------|----------|------|
| manifest `display` 변경 | 없음 | 없음 | 안전 |
| 아이콘 PNG 추가 | 없음 | 없음 | 안전 |
| **SW 교체 (캐싱)** | **높음** | SSE 깨짐, API 꼬임 가능 | `/api/**` network-only 필수 |
| SW 등록 복원 | 낮음 | 없음 | SW 파일과 동시 배포 |
| InstallPrompt 활성화 | 낮음 | 설치 배너 1회 노출 | 기존 유저 전원에게 1회 |
| .htaccess | 없음 | 없음 | 수정 불필요 |

## 권장 배포 순서 (유저 영향 최소화)

1. **1차 배포 (무영향):** 아이콘 PNG 파일 추가 + manifest `display: "standalone"` 변경
2. **2차 배포 (핵심):** `sw.js`를 실제 캐싱 SW로 교체 + 등록 코드 복원 — 반드시 `/api/**` 패턴을 network-only로 설정한 상태에서
3. **3차 배포 (선택):** `InstallPrompt` 주석 해제 (설치 배너 활성화)

이렇게 하면 2차에서 문제가 생기더라도 `sw.js`만 현재 해제 버전으로 롤백하면 즉시 복구됩니다. 1차·3차는 독립적으로 롤백 가능합니다.

---

## 핵심 결론

**가장 주의할 것은 `sw.js` 교체 단 하나입니다.** SSE 스트리밍 + API 경로를 반드시 network-only로 빠뜨리지 않으면 기존 40명 유저의 챗봇·검색·로그인이 깨질 수 있습니다. 나머지 작업은 유저에게 실질적 영향이 없습니다.

---

## SW 교체 시 network-only 필수 경로

```
/api/**              → 모든 API (인증, 기사, 검색, 관리자 등)
/version.json        → 자동 버전 체크 (useVersionCheck)
```

### SSE 스트리밍 엔드포인트 (특히 주의)

```
/api/article-chat.php      → 기사 챗봇
/api/search-analysis.php   → 지능형 검색 종합 분석
/api/admin/chat-stream.php → 관리자 채팅
```
